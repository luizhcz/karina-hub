Visão Geral

O EFS AI Hub é uma plataforma de orquestração de Agentes e Workflows de IA sobre o Microsoft Agent Framework. Ele expõe uma API de chat em tempo real baseada no protocolo AG-UI — compatível tanto com clientes customizados EFS quanto com clientes padrão AG-UI (CopilotKit, AG-UI SDK, etc.).

Este portal documenta o protocolo AG-UI em detalhe e os endpoints de leitura (GET / LIST) de Agentes e Workflows. Para criar, atualizar ou excluir recursos, consulte o Swagger UI.

AG-UI em 60 segundos

AG-UI Protocol é uma especificação aberta (mantida pelo time CopilotKit) que padroniza como uma UI conversa com um agente / workflow de IA. Em vez de cada produto inventar eventos próprios pra streaming de tokens, tool calls, aprovações humanas e atualização de estado, o AG-UI define 17 tipos de evento canônicos que cobrem o ciclo de vida de uma conversa ponta a ponta. Esse portal documenta a implementação do AG-UI no EFS AI Hub.

O que você ganha

CapacidadeComo o AG-UI entrega
Streaming de texto token a token TEXT_MESSAGE_START → vários TEXT_MESSAGE_CONTENT com delta → TEXT_MESSAGE_END
Tool calls (function calling) TOOL_CALL_START → TOOL_CALL_ARGS (streaming) → TOOL_CALL_END → TOOL_CALL_RESULT
Aprovação humana (HITL) Mesma sequência de tool call usando toolCallName: "request_approval" — o stream fica aberto esperando o humano clicar.
Estado compartilhado UI ↔ agente STATE_SNAPSHOT (estado completo) e STATE_DELTA (JSON Patch RFC 6902) — sincroniza form, posição, contexto, etc.
Reconexão sem perder mensagens Cada evento tem id: incremental no SSE; cliente envia Last-Event-ID e o servidor faz replay.
Eventos de domínio fora do schema CUSTOM com customName + customValue arbitrário — extensão idiomática.

Como ele se compara ao que você já conhece

ConceitoOpenAI Chat StreamingAG-UI
Transporte SSE (text/event-stream) SSE (text/event-stream) — mesmo formato wire
Streaming de tokens chat.completion.chunk com choices[].delta.content Eventos discretos: TEXT_MESSAGE_START / CONTENT / END
Tool call tool_calls[] dentro do chunk Eventos próprios: TOOL_CALL_* (start/args streamed/end/result)
HITL (aprovação humana) Não tem — sua app implementa por fora Faz parte do protocolo: tool call request_approval + endpoint /resolve-hitl
Estado compartilhado Não tem STATE_SNAPSHOT + STATE_DELTA via JSON Patch
Reconexão Sem replay — recomeça do zero Last-Event-ID + replay automático
Mental model rápido: pense no AG-UI como "OpenAI streaming + tools + HITL + state + reconnect, tudo num único stream SSE com eventos discretos e tipados".

O que NÃO é AG-UI

  • Não é um SDK proprietário do EFS — é uma spec aberta implementada por vários frameworks (CopilotKit, Microsoft Agent Framework, Mastra, Pydantic AI, LangGraph bridge).
  • Não é REST — é um stream contínuo. Uma única request POST /api/chat/ag-ui/stream recebe N eventos até o turn terminar.
  • Não é WebSocket — é SSE (HTTP unidirecional servidor→cliente). Para enviar mensagens novas o cliente faz POST normal.
  • Não substitui o Swagger — endpoints CRUD de agentes/workflows continuam REST tradicional.

Glossário rápido

TermoSignificado
ThreadConversa contínua. Identificada por threadId; persiste mensagens entre turns.
RunUma execução do workflow disparada por um turn (uma mensagem do usuário). runId identifica essa execução específica.
StepUm nó do workflow que produz fala visível ao usuário (geralmente um agente LLM). Marca-se com STEP_STARTED/STEP_FINISHED.
DeltaFragmento incremental — pedaço de texto (token), pedaço de args de tool call, ou JSON Patch de estado.
SnapshotEstado completo enviado de uma vez (vs delta). Usado em STATE_SNAPSHOT e MESSAGES_SNAPSHOT.
Tool callWorkflow chamando uma função (function calling do LLM ou aprovação humana).
HITLHuman-in-the-loop. Pausa do workflow esperando intervenção humana. Modelado como tool call especial request_approval.
Frame SSEBloco de linhas data: + id: separadas por linha em branco. Um frame = um evento.

Autenticação e Identidade

Todos os endpoints AG-UI são públicos — não exigem token. Mas exigem um header de identidade.

HeaderTipo de usuárioExemplo
x-efs-account Cliente (conta de investimento) x-efs-account: 123456789
x-efs-user-profile-id Assessor de investimentos x-efs-user-profile-id: P-00123
Envie apenas um dos dois headers por requisição. Sem header, o usuário é identificado como anonymous e pode ter funcionalidades restritas dependendo do workflow.

Endpoints AG-UI

Todos os endpoints estão sob o prefixo /api/chat/ag-ui/.

POST /api/chat/ag-ui/stream Envia mensagem e abre o stream SSE ▾

Envia mensagem via array messages[] e abre o stream SSE de resposta.

{

  "threadId": null,

Cancela o run identificado por executionId. O stream original receberá o evento RUN_ERROR com errorCode: "CANCELLED".

O executionId é retornado no evento RUN_STARTED.runId.

Resolve uma aprovação pendente. O stream original continua após receber a resposta — não feche o stream antes.

// Ao reconectar:

    'x-efs-account': conta,
  }
})

Catálogo de Eventos

Cada evento SSE chega no formato data: {json} com ao menos o campo type. Todo evento inclui também timestamp (ISO-8601 UTC) e a linha id: do SSE corresponde ao BusSequenceId usado pelo header Last-Event-ID na reconexão. Esses campos foram omitidos das colunas pra manter a tabela enxuta.

Eventos marcados com terminal encerram o stream — após emiti-los, o servidor fecha a conexão SSE e o cliente não deve esperar mais nada.

Tipo Categoria Campos principais Descrição
RUN_STARTED Lifecycle runId, threadId Workflow iniciou. Salve o threadId para próximas mensagens.
RUN_FINISHED terminal Lifecycle runId, threadId, output Workflow concluiu com sucesso. Stream encerrado após este evento.
RUN_ERROR terminal Lifecycle error, errorCode, runId Workflow falhou ou foi cancelado. Veja a tabela completa de errorCode abaixo.
STEP_STARTED Steps stepId, stepName Agente LLM iniciou execução. Apenas para nós tipo agente — executores de código emitem CUSTOM executor.lifecycle (ver abaixo).
STEP_FINISHED Steps stepId, stepName Agente LLM concluiu. Para outputs não-streamed, vem seguido de TEXT_MESSAGE_START/CONTENT/END reconstruindo a fala.
TEXT_MESSAGE_START Texto messageId, role (default "assistant") Nova mensagem do assistente começando. Crie a bolha de resposta.
TEXT_MESSAGE_CONTENT Texto messageId, delta Fragmento de token. Concatene delta para montar a mensagem.
TEXT_MESSAGE_END Texto messageId Mensagem do assistente finalizada.
TOOL_CALL_START Tools toolCallId, toolCallName, parentMessageId (opcional) Workflow chamando uma tool. Se toolCallName = "request_approval": fluxo HITL.
TOOL_CALL_ARGS Tools toolCallId, delta Argumentos da tool em streaming (fragmentos de JSON). Acumule e parse só ao TOOL_CALL_END.
TOOL_CALL_END Tools toolCallId Fim do streaming de argumentos. Para HITL: mostre os botões de aprovação agora.
TOOL_CALL_RESULT Tools toolCallId, result, messageId, role: "tool" Resultado da tool call. Para HITL: result contém a resposta humana (ex: "Aprovar").
STATE_SNAPSHOT Estado snapshot Estado completo atual do workflow. Emitido no início do stream ou na reconexão.
STATE_DELTA Estado delta Diff incremental de estado em formato JSON Patch (RFC 6902). Aplique sobre o último STATE_SNAPSHOT.
MESSAGES_SNAPSHOT Reconexão messages[] Histórico completo da conversa. Enviado apenas na reconexão sem Last-Event-ID.
CUSTOM Custom customName, customValue Eventos de domínio específicos. customName reconhecidos: ESCALATION, executor.lifecycle (ver tabela abaixo). Outros nomes podem aparecer; consumidores devem ignorar silenciosamente os desconhecidos.
SAFETY_VIOLATION terminal Safety error, errorCode, customValue, runId Conteúdo violou guardrail (Blocklist). errorCode = categoria (ex: PII, SECRETS, FINANCIAL, INTERNAL). Stream encerrado. customValue traz {violationId, phase, retryable}. Veja o fluxo SAFETY_VIOLATION abaixo.

Anatomia de um turno

A tabela acima lista os tipos de evento, mas o que importa pra implementar um cliente é a sequência. Cada turno (uma mensagem do usuário) gera entre 5 e 50+ eventos conforme o caminho. Os 3 caminhos principais cobrem ~95% dos casos:

1. Happy path — agente respondendo só com texto

// Cliente envia: "Qual é o saldo da minha conta?"
// POST /api/chat/ag-ui/stream → 200 text/event-stream

stream ← RUN_STARTED          { runId: "run-abc", threadId: "thr-001" }
stream ← STATE_SNAPSHOT       { snapshot: { ... } }            // estado inicial do thread
stream ← STEP_STARTED         { stepId: "agent-saldo", stepName: "agent-saldo" }
stream ← TEXT_MESSAGE_START   { messageId: "msg-1", role: "assistant" }
stream ← TEXT_MESSAGE_CONTENT { messageId: "msg-1", delta: "Seu " }
stream ← TEXT_MESSAGE_CONTENT { messageId: "msg-1", delta: "saldo " }
stream ← TEXT_MESSAGE_CONTENT { messageId: "msg-1", delta: "é R$ 12.480,33." }
stream ← TEXT_MESSAGE_END     { messageId: "msg-1" }
stream ← STEP_FINISHED        { stepId: "agent-saldo" }
stream ← RUN_FINISHED         { runId: "run-abc", output: "Seu saldo é R$ 12.480,33." }

// stream encerra (servidor fecha SSE)

2. Com tool call (function calling)

// Cliente envia: "Compre 100 PETR4 a 30,00"

stream ← RUN_STARTED          { runId: "run-def", threadId: "thr-002" }
stream ← STEP_STARTED         { stepId: "agent-boleta" }

stream ← TOOL_CALL_START      { toolCallId: "tc-1", toolCallName: "SendOrder" }
stream ← TOOL_CALL_ARGS       { toolCallId: "tc-1", delta: '{"ticker":' }    // args streamed em fragmentos
stream ← TOOL_CALL_ARGS       { toolCallId: "tc-1", delta: '"PETR4","quantity":100}' }
stream ← TOOL_CALL_END        { toolCallId: "tc-1" }                    // fim do streaming de args
stream ← TOOL_CALL_RESULT     { toolCallId: "tc-1", result: "order-789 enviada", role: "tool" }

// agente continua produzindo a resposta final pro usuário
stream ← TEXT_MESSAGE_START   { messageId: "msg-2", role: "assistant" }
stream ← TEXT_MESSAGE_CONTENT { messageId: "msg-2", delta: "Ordem enviada com sucesso..." }
stream ← TEXT_MESSAGE_END     { messageId: "msg-2" }
stream ← STEP_FINISHED        { stepId: "agent-boleta" }
stream ← RUN_FINISHED         { ... }
Importante: os delta de TOOL_CALL_ARGS chegam fragmentados — o JSON pode ser cortado no meio de uma string. Acumule todos os delta num buffer e faça JSON.parse() apenas após TOOL_CALL_END.

3. Com HITL (aprovação humana)

HITL é uma tool call especial — toolCallName: "request_approval". A diferença é que o stream fica aberto esperando o cliente chamar POST /resolve-hitl. Veja o fluxo HITL completo com detalhes do customValue e dos botões.

4. Fim do turno: 3 caminhos terminais

Todo turno termina por exatamente um destes 3 eventos. Após eles, o servidor fecha o SSE e o cliente não deve esperar mais nada:

Evento terminalQuandoO que fazer
RUN_FINISHED success Workflow concluiu sem erro. Habilitar input pro próximo turn. Salvar threadId.
RUN_ERROR erro Falha técnica (timeout, exception, cancelado). Mostrar mensagem de erro. Olhar errorCode pra decidir retry — ver códigos.
SAFETY_VIOLATION bloqueio Conteúdo violou guardrail (PII, secrets, etc). Mostrar UI específica de bloqueio. Não retentar. Ver fluxo.

Cliente mínimo (JavaScript)

Implementação completa em ~70 linhas — fetch SSE + parser linha-a-linha + acumulador de tokens + handler dos 3 caminhos terminais. Roda em browser (com Web Streams API) ou Node 18+. 

// Estado mantido entre turns: persiste threadId pra continuar a mesma conversa.
let threadId = null
const messages = []  // histórico local

async function sendTurn(userText) {
  messages.push({ id: crypto.randomUUID(), role: 'user', content: userText })

  const res = await fetch('/api/chat/ag-ui/stream', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'x-efs-account': '123456789',  // header de identidade — escolha um (ver Autenticação)
    },
    body: JSON.stringify({
      messages,
      threadId,                       // null no primeiro turn, reuso depois
      runId: crypto.randomUUID(),
      tools: [],
      state: {},
    }),
  })

  if (!res.ok) throw new Error(`HTTP ${res.status}`)

  // Estado por turn
  const tokens = new Map()              // messageId → texto acumulado
  const toolArgs = new Map()            // toolCallId → buffer JSON
  let lastEventId = null

  // Parser SSE: lê o body em chunks UTF-8 e quebra por linha em branco (\n\n).
  const reader = res.body.getReader()
  const decoder = new TextDecoder()
  let buffer = ''

  while (true) {
    const { value, done } = await reader.read()
    if (done) break
    buffer += decoder.decode(value, { stream: true })

    const frames = buffer.split('\n\n')
    buffer = frames.pop()  // último pedaço pode estar incompleto

    for (const frame of frames) {
      const lines = frame.split('\n')
      let dataLine = null
      for (const line of lines) {
        if (line.startsWith('data: ')) dataLine = line.slice(6)
        else if (line.startsWith('id: ')) lastEventId = line.slice(4)
        // linhas começando com `:` são heartbeat — ignorar
      }
      if (!dataLine) continue
      const evt = JSON.parse(dataLine)

      switch (evt.type) {
        case 'RUN_STARTED':
          threadId = evt.threadId  // guardar pro próximo turn
          break

        case 'TEXT_MESSAGE_START':
          tokens.set(evt.messageId, '')
          break

        case 'TEXT_MESSAGE_CONTENT':
          tokens.set(evt.messageId, (tokens.get(evt.messageId) ?? '') + (evt.delta ?? ''))
          renderBubble(evt.messageId, tokens.get(evt.messageId))  // streaming UI
          break

        case 'TEXT_MESSAGE_END':
          messages.push({ id: evt.messageId, role: 'assistant', content: tokens.get(evt.messageId) })
          break

        case 'TOOL_CALL_START':
          toolArgs.set(evt.toolCallId, '')
          break

        case 'TOOL_CALL_ARGS':
          toolArgs.set(evt.toolCallId, (toolArgs.get(evt.toolCallId) ?? '') + (evt.delta ?? ''))
          break

        case 'TOOL_CALL_END': {
          const argsJson = toolArgs.get(evt.toolCallId) ?? '{}'
          const args = JSON.parse(argsJson)  // agora é seguro parse
          renderToolCall(evt.toolCallId, args)
          toolArgs.delete(evt.toolCallId)
          break
        }

        case 'STATE_SNAPSHOT':
          replaceState(evt.snapshot)
          break

        case 'STATE_DELTA':
          applyJsonPatch(evt.delta)  // fast-json-patch ou similar
          break

        case 'CUSTOM':
          // Discrimine por customName e ignore desconhecidos.
          if (evt.customName === 'executor.lifecycle') {
            // detalhe técnico — não criar bubble; loggar se útil pra observability
          } else if (evt.customName === 'actor.persisted') {
            // turn robot — backend persistiu sem disparar workflow; MESSAGES_SNAPSHOT a seguir tem histórico atualizado
          }
          break

        case 'RUN_FINISHED':
        case 'RUN_ERROR':
        case 'SAFETY_VIOLATION':
          // 3 caminhos terminais — habilitar input do próximo turn
          handleTurnEnd(evt)
          return
      }
    }
  }
}
Por que fetch + parser manual em vez de EventSource? A API EventSource do browser só faz GET — o AG-UI precisa POST com body. Com fetch você lê o stream do res.body.getReader() e parseia frames SSE manualmente. É ~20 linhas a mais e dá controle total sobre headers, cancelamento e backpressure.

Reconexão

Quando o stream cai (rede, refresh, timeout), guarde o último id: recebido (lastEventId no exemplo) e chame: 

const res = await fetch(`/api/chat/ag-ui/reconnect/${runId}`, {
  headers: {
    'Last-Event-ID': lastEventId,
    'x-thread-id': threadId,
    'x-efs-account': '123456789',
  },
})
// O servidor faz replay dos eventos perdidos. Se Last-Event-ID for null, manda
// MESSAGES_SNAPSHOT + STATE_SNAPSHOT pra resync completo antes de continuar.

Fluxo HITL — Human in the Loop

O HITL ocorre quando o workflow precisa de aprovação humana antes de prosseguir. O stream permanece aberto aguardando a resposta.

// 1. Workflow solicita aprovação stream ← TOOL_CALL_START { toolCallId: "tc-1", toolCallName: "request_approval" } stream ← TOOL_CALL_ARGS { toolCallId: "tc-1", delta: '{"question":"Confirma transferência de R$ 50k?","options":["Aprovar","Rejeitar"]}' } stream ← TOOL_CALL_END { toolCallId: "tc-1" } [ Frontend acumula delta e exibe botões de aprovação — stream permanece aberto ] // 2. Usuário clica em "Aprovar" cliente → POST /api/chat/ag-ui/resolve-hitl { toolCallId: "tc-1", response: "Aprovar" } // 3. Workflow continua stream ← TOOL_CALL_RESULT { toolCallId: "tc-1", result: "Aprovar" } stream ← TEXT_MESSAGE_START stream ← TEXT_MESSAGE_CONTENT [ tokens da resposta final ] stream ← RUN_FINISHED
Dica: Acumule todos os TOOL_CALL_ARGS.delta antes do TOOL_CALL_END e só então parse o JSON — os args podem chegar em múltiplos fragmentos.

Estrutura do delta HITL

{
  "interactionId": "hitl-001",
  "question": "Confirma transferência de R$ 50.000,00 para conta 123456?",
  "options": ["Aprovar", "Rejeitar"],
  "timeoutSeconds": 300
}

Quando options é null, assuma Aprovar / Rejeitar com valores "approved" / "rejected".

Custom Events — extensões idiomáticas AG-UI

O evento CUSTOM é a forma spec-conforme de carregar metadata fora dos 17 tipos canônicos do AG-UI Protocol. O backend usa customName como discriminador. Consumidores devem ignorar silenciosamente nomes desconhecidos (forward-compatible).

customName Quando é emitido Shape de customValue Comportamento esperado da UI
ESCALATION Workflow disparou regra de escalação (operador humano necessário). Variável conforme regra (categoria, contexto, alvo). Renderizar banner / notificar canal de operadores.
executor.lifecycle Início ou fim de um nó executor (código puro, sem LLM). Substitui STEP_STARTED/STEP_FINISHED para esse tipo de nó por design — executores não são "falas" no chat. { nodeId, phase: "started"\|"finished", functionName?, durationMs? } Chat principal: ignorar (sem bubble). Timeline técnica / DevPortal: registrar como linha de progresso usando nodeId e functionName.
actor.persisted Mensagem com actor=robot foi persistida sem disparar workflow (short-circuit). Emitido no SSE sintético do turn robot. Ver seção Actor extension. { messageId, actor: "robot" } Chat principal: marcar a mensagem como "registrada sem execução" (opcional). Cliente que ignora o evento ainda recebe RUN_FINISHED e fecha o turn corretamente.

Por que executor não é STEP_*?

A spec AG-UI define STEP_STARTED/STEP_FINISHED como ciclo de vida de unidades de progresso visíveis ao usuário (1 step ≈ 1 fala). Executores de código (validação, transformação, unwrap de envelopes) são detalhe de implementação do graph — emiti-los como STEP_* gerava bubbles fake no chat com nomes técnicos (post_processor, unwrap_aprovacao_to_ativo) confundindo o usuário.

O caminho via CUSTOM mantém o protocolo canônico (consumidores AG-UI padrão como CopilotKit React SDK continuam funcionando) e ainda dá visibilidade técnica para quem precisa (DevPortal, observability dashboards).

// Exemplo: workflow `atendimento-assessor` durante uma execução
stream ← STEP_STARTED            { stepId: "router-atendimento-assessor", stepName: "router-atendimento-assessor" }
stream ← TEXT_MESSAGE_START      ...
stream ← TEXT_MESSAGE_CONTENT    ...
stream ← TEXT_MESSAGE_END        ...
stream ← STEP_FINISHED           { stepId: "router-atendimento-assessor", ... }

// Próximo nó é executor — sem STEP_*, sem bubble fake
stream ← CUSTOM                  { customName: "executor.lifecycle", customValue: { nodeId: "post_processor", phase: "started", functionName: "service_post_processor" } }
stream ← CUSTOM                  { customName: "executor.lifecycle", customValue: { nodeId: "post_processor", phase: "finished", functionName: "service_post_processor" } }

Extensão actor — registrar postagens de robô sem disparar workflow

Extensão proprietária aditiva à spec AG-UI. Não fere o protocolo — role permanece nos 5 canônicos, e clientes AG-UI padrão (CopilotKit, Mastra, Pydantic AI) ignoram o campo silenciosamente. Permite que frontend ou automação (RPA, sistema externo) poste mensagens na conversa que entram no histórico sem chamar LLM nem consumir budget.

Quando usar

Caso canônico — frontend-mediated function calling: agente LLM retorna um objeto estruturado, o frontend chama um backend EXTERNO (não EFS AI Hub) com a auth dele, recebe o resultado, e posta como actor=robot pra registrar a ação na conversa. No turno seguinte, o LLM vê o resultado no histórico e age coerentemente.

actor Significado Comportamento backend
null / ausente Default — humano Workflow dispara normalmente
"human" Explicitamente humano Idem (equivalente a omitir)
"robot" Postagem programática Persiste sem disparar workflow. SSE sintético com 4 eventos canônicos.

Sequência SSE no turn robot

// Cliente envia:
cliente → POST /api/chat/ag-ui/stream
              {
                "messages": [
                  ...histórico...,
                  { "role": "user", "content": "{\"saldo\":12480.33}", "actor": "robot" }
                ]
              }

// Backend responde com SSE sintético em ~ms (sem LLM, sem cobrança):
stream ← STATE_SNAPSHOT      { snapshot: { ... } }
stream ← RUN_STARTED         { runId, threadId }
stream ← CUSTOM              { customName: "actor.persisted", customValue: { messageId, actor: "robot" } }
stream ← MESSAGES_SNAPSHOT   { messages: [...histórico atualizado...] }
stream ← RUN_FINISHED        { runId, threadId, output: "" }

Validações de payload (400)

  • actor com valor fora de {null, "human", "robot"} (após trim) — mensagem de erro identifica a posição.
  • actor="robot" em mensagem que não é a última do batch — robot fecha turno por design.
  • actor="robot" quando há HITL pendente na execução ativa — HITL programático deve usar POST /api/chat/ag-ui/resolve-hitl.

Trust model — sem auth no backend

O backend confia no campo actor sem validação — o trust boundary fica na camada de proxy upstream. Cliente que conseguir falar diretamente com o backend (sem passar pelo proxy) bypassa cobrança e polui audit. Mitigação operacional, não aplicacional.

Razões da decisão (e quando promover pra v4 com JWT scope) em ADR 0014.

Cliente — handler de CUSTOM actor.persisted

case 'CUSTOM':
  if (evtData.customName === 'actor.persisted') {
    // Opcional: marcar a mensagem como "registrada sem execução"
    // no UI. Mesmo ignorando, RUN_FINISHED encerra o turn corretamente.
    const v = evtData.customValue ?? {}
    console.debug('[robot]', v.messageId)
  }
  break

Quando NÃO usar actor=robot

  • Resposta a HITL pendente — use POST /resolve-hitl. actor=robot + HITL pendente retorna 400.
  • Mensagem do usuário humano sendo digitada na UI — mesmo que vinda de formulário automático (autocomplete, etc), continua sendo humano. Robot é especificamente postagem programática que não representa fala humana.
  • Tool calls do LLM — fluxo TOOL_CALL_* existe pra isso e é dirigido pelo agente. Robot é o oposto: frontend-driven, não LLM-driven.

Patterns comuns — cheat sheet

Receitas curtas pros padrões que todo cliente AG-UI implementa. Use junto com o cliente mínimo.

1. Acumular tokens de TEXT_MESSAGE_CONTENT

Cada delta é um pedaço — concatene por messageId:

const tokens = new Map()  // messageId → string

case 'TEXT_MESSAGE_START':
  tokens.set(evt.messageId, '')
  break

case 'TEXT_MESSAGE_CONTENT':
  tokens.set(evt.messageId, tokens.get(evt.messageId) + evt.delta)
  ui.update(evt.messageId, tokens.get(evt.messageId))   // re-render bubble
  break

case 'TEXT_MESSAGE_END':
  ui.markComplete(evt.messageId)
  break

2. Acumular args de TOOL_CALL_ARGS antes do parse

Nunca faça JSON.parse(delta) — o JSON pode ter sido cortado no meio:

const buffers = new Map()

case 'TOOL_CALL_START':
  buffers.set(evt.toolCallId, '')
  break

case 'TOOL_CALL_ARGS':
  buffers.set(evt.toolCallId, buffers.get(evt.toolCallId) + evt.delta)
  break

case 'TOOL_CALL_END': {
  const args = JSON.parse(buffers.get(evt.toolCallId) ?? '{}')  // agora é seguro
  buffers.delete(evt.toolCallId)
  ...
}

3. Aplicar STATE_DELTA via JSON Patch

O delta é uma lista de operações RFC 6902. Use uma lib como fast-json-patch:

import { applyPatch } from 'fast-json-patch'

let state = {}

case 'STATE_SNAPSHOT':
  state = evt.snapshot     // substitui inteiro
  break

case 'STATE_DELTA':
  state = applyPatch(state, evt.delta).newDocument
  break

4. Detectar fim do turno (3 caminhos terminais)

O cliente precisa tratar os 3 — se você só observar RUN_FINISHED, vai travar a UI em erro/bloqueio:

function handleTurnEnd(evt) {
  switch (evt.type) {
    case 'RUN_FINISHED':
      ui.enableInput()
      break

    case 'RUN_ERROR':
      // errorCode é estável — use ele pra decidir, não a string `error`.
      ui.showError(evt.error)
      if (evt.errorCode === 'BUDGET_EXCEEDED' || evt.errorCode === 'RATE_LIMIT') {
        ui.disableInputUntil(/* política da app */)
      } else {
        ui.enableInput()  // permite retry
      }
      break

    case 'SAFETY_VIOLATION':
      // terminal de bloqueio — NÃO retentar com o mesmo conteúdo.
      ui.showSafetyBlock(evt.errorCode, evt.error)
      ui.enableInput()
      break
  }
}

5. Reagir a tool call vs HITL no TOOL_CALL_START

HITL é uma tool call com toolCallName: "request_approval". Distinga cedo pra mostrar a UI certa:

const hitlPending = new Set()

case 'TOOL_CALL_START':
  if (evt.toolCallName === 'request_approval') {
    hitlPending.add(evt.toolCallId)
  } else {
    ui.renderToolCallSpinner(evt.toolCallId, evt.toolCallName)
  }
  break

case 'TOOL_CALL_END':
  if (hitlPending.has(evt.toolCallId)) {
    const args = JSON.parse(buffers.get(evt.toolCallId) ?? '{}')
    // args = { interactionId, question, options, timeoutSeconds, interactionType }
    ui.showApprovalButtons(evt.toolCallId, args)
    hitlPending.delete(evt.toolCallId)
  }
  break

6. Continuar conversa entre turns

threadId é a chave da conversa contínua — guarde do RUN_STARTED e reuse:

let threadId = null  // global ou em store da app

case 'RUN_STARTED':
  threadId = evt.threadId    // no primeiro turn vem novo; nos seguintes vem o mesmo
  break

// próxima request:
fetch('/api/chat/ag-ui/stream', {
  method: 'POST',
  body: JSON.stringify({ messages: [...], threadId, runId: ... }),
})

7. Parser SSE robusto

O frame SSE só está completo quando você vê uma linha em branco (\n\n). Trate buffer parcial:

let buffer = ''

while (true) {
  const { value, done } = await reader.read()
  if (done) break
  buffer += decoder.decode(value, { stream: true })

  const frames = buffer.split('\n\n')
  buffer = frames.pop()        // último pode estar incompleto — guarda pra próxima iter

  for (const frame of frames) {
    // frame = bloco de linhas: data:..., id:..., :heartbeat, etc.
    let dataLine = null
    let idLine = null
    for (const line of frame.split('\n')) {
      if (line.startsWith('data: ')) dataLine = line.slice(6)
      else if (line.startsWith('id: ')) idLine = line.slice(4)
      // linhas começando com `:` são heartbeat — ignore
    }
    if (!dataLine) continue
    handleEvent(JSON.parse(dataLine), idLine)
  }
}
Pegadinhas comuns:
  • JSON.parse em delta de TOOL_CALL_ARGS chunkado — sempre parta por TOOL_CALL_END.
  • Esquecer de tratar SAFETY_VIOLATION — UI fica travada esperando RUN_FINISHED.
  • Não persistir threadId — backend cria thread nova a cada request, perdendo histórico.
  • Usar EventSource nativo — não suporta POST nem headers customizados; use fetch.
  • Reagir a STEP_* de executores — eles emitem CUSTOM executor.lifecycle em vez (ver Custom Events).

Fluxo SAFETY_VIOLATION — Bloqueio de conteúdo

Quando o Blocklist Guardrail detecta padrão proibido (input do usuário ou output do LLM), o stream emite SAFETY_VIOLATION e encerra imediatamente. O cliente deve renderizar uma mensagem específica (não tratar como RUN_ERROR genérico) e não retentar.

// Caminho input: bloqueio antes mesmo de chamar o LLM (zero token gasto) cliente → POST /api/chat/ag-ui/stream { messages: [{ role: "user", content: "meu CPF é 123.456.789-00" }] } stream ← RUN_STARTED stream ← SAFETY_VIOLATION { errorCode: "PII", error: "Conteúdo bloqueado: PII detectada", customValue: { violationId: "v-abc", phase: "input", retryable: false } } // Caminho output: bloqueio durante streaming do LLM (tokens parciais já contados) stream ← RUN_STARTED stream ← TEXT_MESSAGE_START stream ← TEXT_MESSAGE_CONTENT [ tokens parciais antes da detecção ] stream ← SAFETY_VIOLATION { errorCode: "SECRETS", customValue: { phase: "output", retryable: false } }

Categorias padrão de errorCode

errorCodeCobre
PIICPF, CNPJ, dados pessoais
SECRETSAPI keys, tokens, passwords, chaves PEM
FINANCIALNúmeros de cartão (Luhn), valores sensíveis
INTERNALNomes de tools/sistemas internos (built-in dinâmico)
customizadasDefinidas em aihub.blocklist_pattern_groups via SQL ou via tela /projects/:id/blocklist

Estrutura do customValue

{
  "violationId": "viol-abc-123",
  "category": "PII",
  "phase": "input",
  "retryable": false,
  "patternId": "pii.cpf",
  "contextObfuscated": "meu CPF é [REDACTED-len:14]"
}
Importante: retryable: false sempre — a violação é determinística, retentar a mesma mensagem vai bloquear de novo. Para liberar, ou edite a mensagem ou desligue o grupo do projeto em /projects/:id/blocklist.

Códigos de erro (RUN_ERROR.errorCode)

O campo errorCode identifica a causa do RUN_ERROR. Use ele para roteamento de UX (ex: mostrar botão "tentar de novo" só pra falhas transientes do provider).

errorCodeSignificadoRetentar?
TIMEOUTExecução excedeu timeoutSecondsSim (talvez aumentar timeout)
BUDGET_EXCEEDEDTokens ou custo USD ultrapassaram limite do projeto/workflowNão (ajustar limite)
CANCELLEDCancelamento manual (POST /cancel) ou disconnect grace periodSim
LLM_ERRORFalha na chamada ao provider (rede, 5xx)Sim
LLM_RATE_LIMITProvider rejeitou por rate limit (429)Sim com backoff
LLM_CONTENT_FILTERProvider bloqueou por policy interna (Azure/OpenAI safety)Não (editar conteúdo)
TOOL_ERRORTool invocada gerou exceção não tratadaDepende da tool
FRAMEWORK_ERRORErro interno do orquestrador (bug ou config inválida)Não (reportar)
BACK_PRESSURE_REJECTEDSobrecarga — slot registry recusou nova execuçãoSim com backoff
AGENT_LOOP_LIMITAgente excedeu MaxAgentInvocationsNão (ajustar limite)
HITL_REJECTEDHumano rejeitou aprovação inlineNão (ação intencional)
Nota: Bloqueio por guardrail interno (Blocklist) NÃO usa RUN_ERROR — emite SAFETY_VIOLATION separado, com semântica e UI própria.

Agentes

Agentes são os "cérebros" do sistema — cada um encapsula um modelo LLM, um system prompt, ferramentas disponíveis, middlewares de guarda e política de resiliência. Esta seção cobre apenas as rotas de leitura; mutations (POST/PUT/DELETE) ficam no Swagger UI.

Isolamento por projeto — Todos os agentes pertencem a um projectId. Envie o header x-efs-project-id: <id> para isolar recursos por projeto. O valor padrão é "default" quando o header é omitido.

curl http://localhost:5189/api/agents \
