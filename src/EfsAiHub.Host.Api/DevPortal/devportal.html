Visão Geral

O EFS AI Hub é uma plataforma de orquestração de Agentes e Workflows de IA sobre o Microsoft Agent Framework. Ele expõe uma API de chat em tempo real baseada no protocolo AG-UI — compatível tanto com clientes customizados EFS quanto com clientes padrão AG-UI (CopilotKit, AG-UI SDK, etc.).

🔄
Streaming SSE
Respostas token a token via Server-Sent Events
🤝
HITL
Aprovação humana no loop — stream permanece aberto
⚡
AG-UI Padrão
Compatível com CopilotKit e AG-UI SDK

Quick Start

1
Defina o header de identidade
Escolha entre conta de cliente ou ID de assessor. Sem header, o usuário é anonymous.
# Cliente (conta de investimento)
x-efs-account: 011982329

# Assessor
x-efs-user-profile-id: P-00123
2
Envie a primeira mensagem
Use threadId: null para criar uma nova conversa. O servidor retorna o threadId no evento RUN_STARTED.
curl -X POST http://localhost:5189/api/chat/ag-ui/stream \
  -H "Content-Type: application/json" \
  -H "x-efs-account: 011982329" \
  -d '{"messages":[{"role":"user","content":"Olá, qual é o saldo da minha conta?"}],"threadId":null}' \
  --no-buffer
3
Leia o stream SSE
Cada evento chega como data: {json} separado por linha em branco. Acumule os TEXT_MESSAGE_CONTENT para montar a resposta completa.
id: 1
data: {"type":"RUN_STARTED","runId":"exec-abc","threadId":"conv-xyz"}

id: 2
data: {"type":"TEXT_MESSAGE_START","messageId":"msg-1","role":"assistant"}

id: 3
data: {"type":"TEXT_MESSAGE_CONTENT","messageId":"msg-1","delta":"O saldo"}

id: 4
data: {"type":"TEXT_MESSAGE_CONTENT","messageId":"msg-1","delta":" é R$ 15.000"}

id: 5
data: {"type":"RUN_FINISHED","runId":"exec-abc","threadId":"conv-xyz"}

Autenticação e Identidade

Todos os endpoints AG-UI são públicos — não exigem token. Mas exigem um header de identidade.

HeaderTipo de usuárioExemplo
x-efs-account Cliente (conta de investimento) x-efs-account: 011982329
x-efs-user-profile-id Assessor de investimentos x-efs-user-profile-id: P-00123
Envie apenas um dos dois headers por requisição. Sem header, o usuário é identificado como anonymous e pode ter funcionalidades restritas dependendo do workflow.

Endpoints AG-UI

Todos os endpoints estão sob o prefixo /api/chat/ag-ui/.

POST /api/chat/ag-ui/stream Envia mensagem e abre o stream SSE ▾

Envia mensagem via array messages[] e abre o stream SSE de resposta.

Headers opcionais (extensões EFS)

HeaderDescrição
x-efs-project-idProjeto do tenant. Default: "default"
x-efs-workflow-idWorkflow a executar. Default: padrão do projeto/userType

Body

{
  "messages": [
    { "id": "msg-1", "role": "user", "content": "Qual é o saldo da minha conta?" }
  ],
  "threadId": null,
  "runId": "run-custom-001",
  "tools": [],
  "state": {}
}

Campos principais

CampoTipoDescrição
messages[]arrayHistórico; o último role=user é a mensagem efetiva
threadIdstring | nullID da conversa. null = criar nova
workflowIdstring | nullAlternativo ao header x-efs-workflow-id; body tem prioridade
runIdstringRun ID fornecido pelo cliente

Códigos HTTP de resposta

CódigoSituação

POST /api/chat/ag-ui/cancel Cancela uma execução em andamento ▾

Cancela o run identificado por executionId. O stream original receberá o evento RUN_ERROR com errorCode: "CANCELLED".

{ "executionId": "exec-abc123" }

O executionId é retornado no evento RUN_STARTED.runId.

POST /api/chat/ag-ui/resolve-hitl Responde a uma aprovação humana (HITL) ▾

Resolve uma aprovação pendente. O stream original continua após receber a resposta — não feche o stream antes.

{
  "toolCallId": "tc-xyz789",
  "response": "approved"
}
CampoValores
toolCallIdID recebido no evento TOOL_CALL_START
response"approved", "rejected", ou texto customizado das options[]
GET /api/chat/ag-ui/reconnect/{executionId} Reconecta após perda de conexão ▾

Reconecta a uma execução em andamento com suporte a replay via Last-Event-ID.

HeaderDescrição
Last-Event-IDSequence ID para replay a partir daquele ponto
x-thread-idID da conversa (threadId)
Sem Last-Event-ID: o servidor envia MESSAGES_SNAPSHOT + STATE_SNAPSHOT para resync completo, depois continua o stream. 
// Armazene o último id: recebido do stream
let lastEventId = null

// Ao reconectar:
const res = await fetch(`/api/chat/ag-ui/reconnect/${executionId}`, {
  headers: {
    'Last-Event-ID': lastEventId,
    'x-thread-id': threadId,
    'x-efs-account': conta,
  }
})

Catálogo de Eventos

Cada evento SSE chega no formato data: {json} com ao menos o campo type. Todo evento inclui também timestamp (ISO-8601 UTC) e a linha id: do SSE corresponde ao BusSequenceId usado pelo header Last-Event-ID na reconexão. Esses campos foram omitidos das colunas pra manter a tabela enxuta.

Eventos marcados com terminal encerram o stream — após emiti-los, o servidor fecha a conexão SSE e o cliente não deve esperar mais nada.

Tipo Categoria Campos principais Descrição
RUN_STARTED Lifecycle runId, threadId Workflow iniciou. Salve o threadId para próximas mensagens.
RUN_FINISHED terminal Lifecycle runId, threadId, output Workflow concluiu com sucesso. Stream encerrado após este evento.
RUN_ERROR terminal Lifecycle error, errorCode, runId Workflow falhou ou foi cancelado. Veja a tabela completa de errorCode abaixo.
STEP_STARTED Steps stepId, stepName Etapa interna do workflow iniciou (agente ou nó do grafo).
STEP_FINISHED Steps stepId, stepName Etapa interna concluiu.
TEXT_MESSAGE_START Texto messageId, role (default "assistant") Nova mensagem do assistente começando. Crie a bolha de resposta.
TEXT_MESSAGE_CONTENT Texto messageId, delta Fragmento de token. Concatene delta para montar a mensagem.
TEXT_MESSAGE_END Texto messageId Mensagem do assistente finalizada.
TOOL_CALL_START Tools toolCallId, toolCallName, parentMessageId (opcional) Workflow chamando uma tool. Se toolCallName = "request_approval": fluxo HITL.
TOOL_CALL_ARGS Tools toolCallId, delta Argumentos da tool em streaming (fragmentos de JSON). Acumule e parse só ao TOOL_CALL_END.
TOOL_CALL_END Tools toolCallId Fim do streaming de argumentos. Para HITL: mostre os botões de aprovação agora.
TOOL_CALL_RESULT Tools toolCallId, result, messageId, role: "tool" Resultado da tool call. Para HITL: result contém a resposta humana (ex: "Aprovar").
STATE_SNAPSHOT Estado snapshot Estado completo atual do workflow. Emitido no início do stream ou na reconexão.
STATE_DELTA Estado delta Diff incremental de estado em formato JSON Patch (RFC 6902). Aplique sobre o último STATE_SNAPSHOT.
MESSAGES_SNAPSHOT Reconexão messages[] Histórico completo da conversa. Enviado apenas na reconexão sem Last-Event-ID.
CUSTOM Custom customName, customValue Eventos de domínio específicos emitidos pelo workflow (ex: customName: "ESCALATION").
SAFETY_VIOLATION terminal Safety error, errorCode, customValue, runId Conteúdo violou guardrail (Blocklist). errorCode = categoria (ex: PII, SECRETS, FINANCIAL, INTERNAL). Stream encerrado. customValue traz {violationId, phase, retryable}. Veja o fluxo SAFETY_VIOLATION abaixo.

Fluxo HITL — Human in the Loop

O HITL ocorre quando o workflow precisa de aprovação humana antes de prosseguir. O stream permanece aberto aguardando a resposta.

// 1. Workflow solicita aprovação stream ← TOOL_CALL_START { toolCallId: "tc-1", toolCallName: "request_approval" } stream ← TOOL_CALL_ARGS { toolCallId: "tc-1", delta: '{"question":"Confirma transferência de R$ 50k?","options":["Aprovar","Rejeitar"]}' } stream ← TOOL_CALL_END { toolCallId: "tc-1" } [ Frontend acumula delta e exibe botões de aprovação — stream permanece aberto ] // 2. Usuário clica em "Aprovar" cliente → POST /api/chat/ag-ui/resolve-hitl { toolCallId: "tc-1", response: "Aprovar" } // 3. Workflow continua stream ← TOOL_CALL_RESULT { toolCallId: "tc-1", result: "Aprovar" } stream ← TEXT_MESSAGE_START stream ← TEXT_MESSAGE_CONTENT [ tokens da resposta final ] stream ← RUN_FINISHED
Dica: Acumule todos os TOOL_CALL_ARGS.delta antes do TOOL_CALL_END e só então parse o JSON — os args podem chegar em múltiplos fragmentos.

Estrutura do delta HITL

{
  "interactionId": "hitl-001",
  "question": "Confirma transferência de R$ 50.000,00 para conta 123456?",
  "options": ["Aprovar", "Rejeitar"],
  "timeoutSeconds": 300
}

Quando options é null, assuma Aprovar / Rejeitar com valores "approved" / "rejected".

Fluxo SAFETY_VIOLATION — Bloqueio de conteúdo

Quando o Blocklist Guardrail detecta padrão proibido (input do usuário ou output do LLM), o stream emite SAFETY_VIOLATION e encerra imediatamente. O cliente deve renderizar uma mensagem específica (não tratar como RUN_ERROR genérico) e não retentar.

// Caminho input: bloqueio antes mesmo de chamar o LLM (zero token gasto) cliente → POST /api/chat/ag-ui/stream { messages: [{ role: "user", content: "meu CPF é 123.456.789-00" }] } stream ← RUN_STARTED stream ← SAFETY_VIOLATION { errorCode: "PII", error: "Conteúdo bloqueado: PII detectada", customValue: { violationId: "v-abc", phase: "input", retryable: false } } // Caminho output: bloqueio durante streaming do LLM (tokens parciais já contados) stream ← RUN_STARTED stream ← TEXT_MESSAGE_START stream ← TEXT_MESSAGE_CONTENT [ tokens parciais antes da detecção ] stream ← SAFETY_VIOLATION { errorCode: "SECRETS", customValue: { phase: "output", retryable: false } }

Categorias padrão de errorCode

errorCodeCobre
PIICPF, CNPJ, dados pessoais
SECRETSAPI keys, tokens, passwords, chaves PEM
FINANCIALNúmeros de cartão (Luhn), valores sensíveis
INTERNALNomes de tools/sistemas internos (built-in dinâmico)
customizadasDefinidas em aihub.blocklist_pattern_groups via SQL ou via tela /projects/:id/blocklist

Estrutura do customValue

{
  "violationId": "viol-abc-123",
  "category": "PII",
  "phase": "input",
  "retryable": false,
  "patternId": "pii.cpf",
  "contextObfuscated": "meu CPF é [REDACTED-len:14]"
}
Importante: retryable: false sempre — a violação é determinística, retentar a mesma mensagem vai bloquear de novo. Para liberar, ou edite a mensagem ou desligue o grupo do projeto em /projects/:id/blocklist.

Códigos de erro (RUN_ERROR.errorCode)

O campo errorCode identifica a causa do RUN_ERROR. Use ele para roteamento de UX (ex: mostrar botão "tentar de novo" só pra falhas transientes do provider).

errorCodeSignificadoRetentar?
TIMEOUTExecução excedeu timeoutSecondsSim (talvez aumentar timeout)
BUDGET_EXCEEDEDTokens ou custo USD ultrapassaram limite do projeto/workflowNão (ajustar limite)
CANCELLEDCancelamento manual (POST /cancel) ou disconnect grace periodSim
LLM_ERRORFalha na chamada ao provider (rede, 5xx)Sim
LLM_RATE_LIMITProvider rejeitou por rate limit (429)Sim com backoff
LLM_CONTENT_FILTERProvider bloqueou por policy interna (Azure/OpenAI safety)Não (editar conteúdo)
TOOL_ERRORTool invocada gerou exceção não tratadaDepende da tool
FRAMEWORK_ERRORErro interno do orquestrador (bug ou config inválida)Não (reportar)
BACK_PRESSURE_REJECTEDSobrecarga — slot registry recusou nova execuçãoSim com backoff
AGENT_LOOP_LIMITAgente excedeu MaxAgentInvocationsNão (ajustar limite)
HITL_REJECTEDHumano rejeitou aprovação inlineNão (ação intencional)
Nota: Bloqueio por guardrail interno (Blocklist) NÃO usa RUN_ERROR — emite SAFETY_VIOLATION separado, com semântica e UI própria.

Agentes

Agentes são os "cérebros" do sistema — cada um encapsula um modelo LLM, um sistema de instruções (system prompt), ferramentas disponíveis, middlewares de guarda e política de resiliência. Eles são referenciados por Workflows, que definem a orquestração entre múltiplos agentes.

Isolamento por projeto — Todos os agentes pertencem a um projectId. Envie o header x-efs-project-id: <id> para isolar recursos por projeto. O valor padrão é "default" quando o header é omitido.
📁
Isolamento por Projeto
Agentes são escopados por projectId — diferentes times não interferem uns nos outros
🔖
Versionamento Automático
Cada edição gera um snapshot imutável com hash — rollback é possível para qualquer revisão anterior
🛠
Tooling & Middlewares
Functions, MCP, web search, code interpreter — mais pipeline de middlewares por agente

GET /api/agents Listar todos os agentes do projeto ▾

Retorna todos os agentes do projeto ativo (definido pelo header x-efs-project-id). Resposta é um array de AgentResponse.

curl http://localhost:5189/api/agents \
  -H "x-efs-account: 011982329"
Para buscar agentes de um projeto específico: -H "x-efs-project-id: meu-projeto"
GET /api/agents/{id} Buscar agente por ID ▾

Retorna o agente completo com todos os campos.

curl http://localhost:5189/api/agents/assessor-investimentos \
  -H "x-efs-account: 011982329"
CódigoSituação
200Agente encontrado
404ID não existe
PUT /api/agents/{id} Atualizar agente (gera nova versão) ▾

Atualiza o agente e cria automaticamente uma nova AgentVersion com snapshot imutável. O body tem o mesmo schema do POST.

curl -X PUT http://localhost:5189/api/agents/assessor-investimentos \
  -H "Content-Type: application/json" \
  -H "x-efs-account: 011982329" \
  -d '{
    "id": "assessor-investimentos",
    "name": "Assessor de Investimentos v2",
    "model": { "deploymentName": "gpt-4o", "temperature": 0.1 },
    "instructions": "Instrução atualizada..."
  }'
O PUT sempre gera uma nova revisão — nunca sobrescreve. Use POST /rollback para reverter.
DELETE /api/agents/{id} Remover agente ▾

Remove o agente e todas as suas versões. Retorna 204 No Content.

Atenção: Workflows que referenciam este agente continuarão a existir, mas falharão ao tentar usá-lo. 
curl -X DELETE http://localhost:5189/api/agents/assessor-investimentos \
  -H "x-efs-account: 011982329"

Versionamento de Agentes

Cada chamada PUT /api/agents/{id} cria automaticamente um snapshot imutável (AgentVersion) com contentHash SHA-256 de toda a configuração. Você nunca perde histórico.

// Criação inicial POST /api/agents → revision 1 contentHash: a1b2c3... // Primeira edição (novo modelo) PUT /api/agents/{id} → revision 2 contentHash: d4e5f6... // Segunda edição (novas instruções) PUT /api/agents/{id} → revision 3 contentHash: g7h8i9... // Rollback para revision 1 — cria revision 4 com o conteúdo de revision 1 POST /api/agents/{id}/rollback → revision 4 contentHash: a1b2c3... (igual ao 1)

Listar versões

curl http://localhost:5189/api/agents/assessor-investimentos/versions \
  -H "x-efs-account: 011982329"

Buscar versão específica

curl http://localhost:5189/api/agents/assessor-investimentos/versions/{versionId} \
  -H "x-efs-account: 011982329"

Rollback para versão anterior

curl -X POST http://localhost:5189/api/agents/assessor-investimentos/rollback \
  -H "Content-Type: application/json" \
  -H "x-efs-account: 011982329" \
  -d '{
    "targetVersionId": "uuid-da-versao-alvo",
    "changeReason": "Revertendo instrução problemática do deploy de ontem"
  }'

Exemplo Completo — Criar Agente e Conversar

Fluxo end-to-end: criar agente → criar workflow → criar conversa → streaming.

const BASE = 'http://localhost:5189'
const HEADERS = {
  'Content-Type': 'application/json',
  'x-efs-account': '011982329',

// ── 1. Criar o agente ────────────────────────────────────────────────────
const agentRes = await fetch(`${BASE}/api/agents`, {
  method: 'POST',
  headers: HEADERS,
  body: JSON.stringify({
    id: 'assessor-demo',
    name: 'Assessor Demo',
    model: { deploymentName: 'gpt-4o', temperature: 0.2 },
    instructions: 'Você é um assessor de investimentos. Responda sempre em português.',
    tools: [
      { type: 'function', name: 'get_client_position' },
      { type: 'function', name: 'get_market_data' },
    ],
    middlewares: [{ type: 'AccountGuard', enabled: true }],
    resilience: { maxRetries: 2, initialDelayMs: 500 },
    costBudget: { maxCostUsd: 0.25 },
  }),
})
const agent = await agentRes.json()
console.log('Agente criado:', agent.id, '— revision 1')

// ── 2. Criar o workflow referenciando o agente ───────────────────────────
const wfRes = await fetch(`${BASE}/api/workflows`, {
  method: 'POST',
  headers: HEADERS,
  body: JSON.stringify({
    id: 'wf-assessoria-demo',
    name: 'Workflow Assessoria Demo',
    orchestrationMode: 'Sequential', // Sequential | Concurrent | Handoff | GroupChat | Graph
    agents: [{ agentId: 'assessor-demo' }],
  }),
})
const workflow = await wfRes.json()
console.log('Workflow criado:', workflow.id)

// ── 3. Criar conversa ────────────────────────────────────────────────────
const convRes = await fetch(`${BASE}/api/conversations`, {
  method: 'POST',
  headers: HEADERS,
  body: JSON.stringify({ workflowId: 'wf-assessoria-demo' }),
})
// Resposta: { conversationId, userId, userType, workflowId, createdAt }
const conv = await convRes.json()
const conversationId = conv.conversationId
console.log('Conversa:', conversationId, '| userId:', conv.userId)

// ── 4. Streaming via AG-UI ───────────────────────────────────────────────
const streamRes = await fetch(`${BASE}/api/chat/ag-ui/stream`, {
  method: 'POST',
  headers: { ...HEADERS, 'x-efs-workflow-id': 'wf-assessoria-demo' },
  body: JSON.stringify({
    messages: [{ role: 'user', content: 'Quais são as melhores oportunidades em renda fixa agora?' }],
    threadId: conversationId,
  }),
})

const reader = streamRes.body.getReader()
const decoder = new TextDecoder()
let buffer = ''

while (true) {
  const { value, done } = await reader.read()
  if (done) break

  buffer += decoder.decode(value, { stream: true })
  const frames = buffer.split('\n\n')
  buffer = frames.pop()

  for (const frame of frames) {
    const dataLine = frame.split('\n').find(l => l.startsWith('data: '))
    if (!dataLine) continue
    const evt = JSON.parse(dataLine.slice(6))

    if (evt.type === 'TEXT_MESSAGE_CONTENT') {
      process.stdout.write(evt.content ?? evt.delta)
    } else if (evt.type === 'RUN_FINISHED') {
      console.log('\n[FIM]')
    } else if (evt.type === 'RUN_ERROR') {
      console.error('\n[ERRO]', evt.error, `(${evt.errorCode})`)
    }
  }
}
Dica — fingerprinting de tools: Ao criar o agente, você pode passar fingerprintHash em cada tool do tipo function. Use GET /api/functions para obter o hash atual de cada function e garantir que ela não mudou desde que você testou o agente.

Workflows

Workflows orquestram um ou mais Agentes para resolver tarefas complexas. Cada workflow define quais agentes participam, como se comunicam (modo de orquestração), quando disparam (trigger) e limites de execução (tokens, custo, timeout).

Isolamento por projeto — Assim como os agentes, workflows pertencem a um projectId (header x-efs-project-id). Workflows com visibility: "global" aparecem em todos os projetos via GET /api/workflows/visible.
🔀
5 Modos de Orquestração
Sequential, Concurrent, Handoff, GroupChat, Graph — cada um com semântica própria
⚡
Trigger Flexível
OnDemand (API), Scheduled (cron), EventDriven (webhook) — com suporte a disparo único
🛡
Guardrails de Execução
Limites de tokens, custo, invocações de agente, timeout e concorrência por workflow

Modos de Orquestração

Definido pelo campo orchestrationMode (obrigatório). Determina como os agentes interagem entre si.

Sequential Sequência linear de agentes

Cada agente recebe o output do anterior como input. Ideal para pipelines onde cada etapa enriquece o resultado.

Usuário → Agente A → Agente B → Agente C → Resposta
Concurrent Execução paralela

Todos os agentes recebem o mesmo input e executam simultaneamente. Resultado é a agregação dos outputs. Ideal para análises multidimensionais.

        ┌→ Agente A ─┐
Usuário ─┤→ Agente B ─├→ Resposta agregada
        └→ Agente C ─┘
Handoff Roteamento por escalação

Um agente pode escalar para outro via EmitEscalationSignal. O roteamento é controlado por routingRules[]. Ideal para triage — um agente generalista roteia para especialistas.

Usuário → Triagem → (escala) → Especialista A
                             └→ Especialista B
GroupChat Chat em grupo com manager

Um agente manager coordena múltiplos participants. O manager decide quando encerrar o chat. Ideal para tarefas colaborativas entre especialistas.

Requer exatamente 1 agente com role: "manager" e N agentes com role: "participant".
Graph Grafo dirigido com nós e arestas

Máximo controle — você define edges[] entre agentes e executores (código puro sem LLM). Suporta roteamento condicional, fan-out/fan-in e loops. Veja a seção Graph Mode.

Graph Mode — Arestas e Tipos

No modo Graph, você define o fluxo explicitamente via edges[]. Cada aresta conecta nós (agentes ou executores) com um tipo de conexão específico.

Tipos de Aresta (WorkflowEdgeType)

EdgeTypeComportamentoCampos usados
Direct Conexão 1→1 sem condição. Fluxo sempre passa de from para to. from, to
Conditional Conexão 1→1 com condição. O fluxo passa apenas se o output do nó de origem contém a string condition (case-insensitive). from, to, condition
Switch Ramificação 1→N com casos. O primeiro case.condition encontrado no output vence. Se nenhum combinar, o caso isDefault: true é ativado. from, cases[]
FanOut Fan-out 1→N paralelo. O nó de origem envia para todos os nós em targets[] simultaneamente. from, targets[]
FanIn Fan-in N→1 barreira. Aguarda todos os nós em sources[] antes de prosseguir para to. sources[], to

Campos de WorkflowEdge

CampoTipoDescrição
fromstring?ID do nó de origem
tostring?ID do nó de destino (Direct, Conditional, FanIn)
conditionstring?Substring a encontrar no output do nó de origem (Conditional)
edgeTypestringTipo da aresta. Default: Direct
targetsstring[]IDs dos nós de destino (FanOut)
sourcesstring[]IDs dos nós de origem (FanIn)
casesarrayCasos do switch (Switch). Cada item: { condition, targets[], isDefault }

Exemplo Completo — JavaScript

Implementação completa do cliente AG-UI com suporte a streaming, HITL e cancel.

async function chat(conta, mensagem, threadId = null, workflowId = null) {
  const headers = {
    'Content-Type': 'application/json',
    'x-efs-account': conta,
  }
  if (workflowId) headers['x-efs-workflow-id'] = workflowId

  const res = await fetch('/api/chat/ag-ui/stream', {
    method: 'POST',
    headers,
    body: JSON.stringify({ messages: [{ role: 'user', content: mensagem }], threadId }),
  })

  if (!res.ok) throw new Error(`HTTP ${res.status}`)

  const reader = res.body.getReader()
  const decoder = new TextDecoder()
  let buffer = ''
  let currentThreadId = threadId
  let runId = null

  // Buffers para TOOL_CALL_ARGS — acumule antes de parsear
  const toolArgBuffers = new Map()
  const hitlPending    = new Set()

  while (true) {
    const { value, done } = await reader.read()
    if (done) break

    // Parsing por frame SSE (\n\n) para capturar id: e data:
    buffer += decoder.decode(value, { stream: true })
    const frames = buffer.split('\n\n')
    buffer = frames.pop() // último frame pode estar incompleto

    for (const frame of frames) {
      if (!frame.trim()) continue

      let evtData = null
      for (const line of frame.split('\n')) {
        if (line.startsWith('data: ')) evtData = JSON.parse(line.slice(6))
      }
      if (!evtData) continue

      switch (evtData.type) {
        case 'RUN_STARTED':
          currentThreadId = evtData.threadId
          runId = evtData.runId
          console.log('Conversa:', currentThreadId, '| Run:', runId)
          break

        case 'TEXT_MESSAGE_CONTENT':
          process.stdout.write(evtData.delta)
          break

        case 'STEP_STARTED':
          console.log(`\n⏳ ${evtData.stepName}`)
          break

        case 'STEP_FINISHED':
          console.log(`✓ ${evtData.stepName}`)
          break

        case 'TOOL_CALL_START':
          if (evtData.toolCallName === 'request_approval') {
            hitlPending.add(evtData.toolCallId)
          }
          toolArgBuffers.set(evtData.toolCallId, '')
          break

        case 'TOOL_CALL_ARGS':
          // Acumule — NÃO parse aqui; o JSON pode chegar em fragmentos
          toolArgBuffers.set(
            evtData.toolCallId,
            (toolArgBuffers.get(evtData.toolCallId) ?? '') + (evtData.delta ?? '')
          )
          break

        case 'TOOL_CALL_END':
          if (hitlPending.has(evtData.toolCallId)) {
            hitlPending.delete(evtData.toolCallId)
            const args = JSON.parse(toolArgBuffers.get(evtData.toolCallId) ?? '{}')
            console.log('\n[HITL]', args.question)

            // Mantenha o stream aberto e resolva em paralelo:
            fetch('/api/chat/ag-ui/resolve-hitl', {
              method: 'POST',
              headers: { 'Content-Type': 'application/json' },
              body: JSON.stringify({ toolCallId: evtData.toolCallId, response: 'approved' }),
            })
          }
          toolArgBuffers.delete(evtData.toolCallId)
          break

        case 'RUN_FINISHED':
          console.log('\n[DONE]')
          break

        case 'RUN_ERROR':
          console.error('\n[ERROR]', evtData.error, `(${evtData.errorCode})`)
          break
      }
    }
  }

  return currentThreadId
}

// Cancel durante o streaming:
async function cancelRun(runId) {
  await fetch('/api/chat/ag-ui/cancel', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ executionId: runId }),
  })
}
